The Twin Dragons — One Core, Two Souls

Whether you live in the terminal or prefer a visual interface, Dragonote has you covered. The CLI (dragonote) is a standalone binary you can drop into any CI/CD pipeline or fire from your shell. The Desktop GUI wraps the exact same Rust scanning engine in a Tauri-powered native window with file trees, checkboxes, and a directory picker. Both share identical scanning, export, and truncation logic — same output, every time.

Hoarding Knowledge — Smart Context Scanning

Point Dragonote at any project directory and it automatically understands what matters. Your .gitignore rules are respected out of the box — no configuration needed. Binary files, node_modules, build artifacts, and common junk directories are detected and skipped before a single token is counted.

Test directories (tests/, __tests__/, spec/) are automatically detected and exported separately so your LLM gets clean source context without test noise mixed in. Resource directories like assets/ and static/ are flagged and toggleable.

Pro unlocks .dragignore — a persistent, project-level ignore file that sits alongside your .gitignore and gives you fine-grained control over exactly what the dragon hoards.

Every scan produces a complete set of structured Markdown artifacts — ready to paste directly into any LLM conversation window.

Precise Token Truncation — Go Pro

Every LLM has a context window. Dragonote makes sure your export fits inside it without wasting a single token. The Basic tier uses conservative fixed limits (200k code / 100k test tokens) that work with any modern model.

Pro unlocks full control: set custom token limits, switch to character-based measurement, or use exact token budget mode powered by tiktoken (cl100k_base) to slice your codebase to a precise count. Need to split a massive monorepo across multiple conversations? Auto-split partitions your export into correctly-sized chunks with zero manual math.

Tech Specs

What powers dragoNotebook under the hood.

Core Engine

  • Language: Rust (100% — no garbage collector, no runtime)
  • CLI binary: single native executable, zero dependencies
  • Scanning: walkdir + custom .gitignore parser
  • Token counting: tiktoken-rs (cl100k_base)
  • Parallel mode: rayon (Pro)
  • Archive export: zip crate (Pro)

Desktop GUI

  • Framework: Tauri 2.0
  • Frontend: vanilla HTML / CSS / JS — no Node.js, no bundler
  • Renderer: OS-native WebView (WebView2 on Win, WebKit on macOS/Linux)
  • IPC: typed Rust commands, JSON serialization
  • Dialogs: native OS file pickers via tauri-plugin-dialog

Resource Footprint

  • Idle RAM: ~30–50 MB (no Electron, no Node, no Chrome)
  • Installer size: ~4–8 MB (leverages OS WebView)
  • Startup: under 1 second on modern hardware
  • Background processes: none — no dev server, no daemon
  • Runtime deps: WebView2 (Win 10+, pre-installed), WebKit (macOS/Linux, always available)

Cross-Platform

  • Windows: 10 (1803+) / 11 — NSIS installer, MSI, standalone CLI
  • macOS: 10.15 Catalina+ — DMG, standalone CLI
  • Linux: .deb, .AppImage, standalone CLI
  • Architecture: x86_64 (ARM builds planned)
  • CI-tested: 119 cross-platform unit & integration tests

Why not Electron?

Electron ships an entire Chromium browser and a Node.js runtime inside every app. That means a simple file-scanning utility would carry ~150–300 MB of overhead, spawn multiple background processes, and idle at 100–200 MB of RAM before the user even clicks a button. dragoNotebook targets developers whose machines are already running IDEs, browsers, Docker containers, and language servers — every megabyte of headroom matters.

Tauri sidesteps this entirely. Instead of bundling a browser, it uses the WebView already installed on the operating system — WebView2 on Windows (Chromium-based, pre-installed on Windows 10 21H2+ and all Windows 11 machines), WebKit on macOS and Linux. The result: a ~4–8 MB installer that idles at ~30–50 MB of RAM with zero Node.js processes running in the background.

No Node.js — not even at build time

A common concern with web-based desktop frameworks: “does it require Node.js?” With Tauri, the answer is no — not at runtime, and not in dragoNotebook’s case, not at build time either. The frontend is static HTML, CSS, and vanilla JavaScript. There is no npm install, no node_modules, no webpack, no dev server. The build pipeline is pure cargo. Users who download the app get a native installer; Node.js never enters the picture.

This matters because developer workstations are resource-constrained. A desktop app that silently spins up a Node.js event loop, an HTTP server, and a file watcher just to render a config form is paying an engineering tax that dragoNotebook avoids entirely. No server process, no port binding, no extra RAM eaten by a runtime that exists only to shuttle data between two languages.

Why not a pure-Rust GUI (egui, iced, etc.)?

Pure-Rust immediate-mode frameworks like egui (eframe) compile to a single binary with zero external dependencies — genuinely impressive for certain workloads. But dragoNotebook’s UI is a form-driven configuration tool: file trees, checkboxes, dropdown menus, text inputs, collapsible panels, tier-gated feature visibility, and responsive layout. This is exactly the kind of interface that HTML/CSS was built to express in a few lines and that immediate-mode Rust GUIs require hundreds of lines of manual layout code to replicate.

Where egui shines instead

Pure-Rust frameworks are the better choice when the application is the rendering — real-time data visualization, interactive charts at 60 fps, GPU-accelerated canvases where every frame repaints from computed data with zero serialization overhead. For those workloads, immediate-mode Rust drawing directly on an OpenGL surface has a clear performance advantage over shipping data across an IPC bridge to a browser renderer.

Bottom line: Tauri gives dragoNotebook a polished, accessible, CSS-styled desktop experience with built-in installers and auto-update — at a fraction of Electron’s resource cost and without requiring Node.js at build or runtime. No bundled browser, no background server, no wasted RAM. The trade-off (OS WebView dependency, slightly slower cold start) is well worth it for a form-driven developer tool that prioritizes low overhead and cross-platform reach.
